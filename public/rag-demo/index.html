The pipeline, one step at a time

RAG is four moves. Open “Why it matters” on any card to go deeper at your own pace.

⌗

Chunk

Documents are split into overlapping pieces that fit a retrieval window.

Why it matters
Whole documents are too big to compare precisely. Splitting into small, overlapping chunks means a question can match the exact passage that answers it — and overlap keeps a sentence from being cut off at a boundary.
→
∿

Embed

Each chunk becomes a vector — a point in semantic space.

Why it matters
An embedding turns text into numbers so that related text lands near each other in space. Big models learn deep synonym links; this demo’s offline embedder is lightweight — it scores shared words and phrasing, so keep a key term or two in your question and it finds the right chunk.
→
◎

Retrieve

Your question is embedded too; the closest chunks win.

Why it matters
The question becomes a vector in the same space. We measure cosine similarity to every chunk and keep the top few — those become the evidence. You’ll see the scores below as bars.
→
✦

Generate

An answer is composed strictly from the retrieved context.

Why it matters
The answer is built only from what was retrieved, so it stays grounded in your sources instead of guessing. Add a Claude key below to see a real LLM do this with citations.

From keywords to meaning

The whole trick of RAG is the Embed → Retrieve pair. Here’s the idea behind it.

Old way Keyword search

Matches the exact words you typed. Ask “how does Pikachu grow up?” and a document that only says “Pikachu evolves” may never surface — the words don’t line up.

RAG way Semantic search

Compares similarity, not exact spelling. Both your question and the chunks become vectors, and the closest ones win — so a reworded question still finds the passage that answers it.

💡 Cosine similarity is just the angle between two vectors: small angle → very related → high score. Those are the scores you’ll see on the retrieved chunks in a moment.

🔬 This demo runs a tiny offline embedder so it works with no API key — it matches shared words and phrasing rather than deep synonyms. That’s the lesson: embedder quality is a dial you control. Swap in a production embedding model (one line of code) and the same pipeline understands paraphrases far better.

Your turn — experiment

You learn this fastest by poking at it. Try these and watch what changes:

  • 1Flip Hybrid ↔ Semantic only on the same question and compare. Hybrid adds a keyword (BM25) signal on top of meaning — watch an exact name or rare term climb the ranking.
  • 2Watch the quality bars. Ask something well-covered, then something the docs barely mention — see groundedness and context relevance drop, and check which chunks earn a “✓ cited” badge.
  • 3Paste your own notes into the knowledge base, re-index, and ask about them. RAG works on any text you give it.
  • 4Switch topics — Coffee, Pokédex, or Embeddings — re-index, and notice the pipeline is identical. The only thing that changed is the data.

Use it in your project

import { RAG } from "rag-made-easy";
import { ClaudeGenerator } from "rag-made-easy/claude"; // optional

const rag = new RAG({ generator: new ClaudeGenerator() });

await rag.add([{ id: "handbook", text: docs }]);
const { answer, sources } = await rag.query("What temperature for pour-over?");

Every stage is pluggable: implement Embedder, VectorStore, or Generator to use any model or database.
